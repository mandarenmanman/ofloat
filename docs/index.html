Vibe Coding 的终极
你只需要一句话。剩下的交给ofloat

ofloat 通过将复杂的基础设施语义统一收敛为单一的 HTTP 协议接口，让 AI 只需聚焦纯业务逻辑，从而在最小化上下文占用的同时，最大化生成代码的环境适应性和部署有效性。

Vibe Coding 的问题

  • AI 生成的代码直接 import Redis SDK、Kafka client，业务和基础设施耦合
  • 换个数据库或消息队列，AI 要重写大量代码
  • 本地能跑，部署到生产环境困难重重
  • AI 不懂运维，生成的代码缺乏安全沙箱和资源隔离

ofloat 的解法

  • 业务代码零 SDK 依赖，所有基础设施操作 = 向 localhost:3500 发 HTTP 请求
  • 换 Redis → DynamoDB → Kafka？改一行 YAML，代码不动
  • deploy.ps1 一键部署，编译 → 推送 → 上线全自动
  • WASM 沙箱隔离，微秒级冷启动，几百 KB 内存

两种架构模式

业务代码与基础设施完全解耦，按场景选择最合适的模式。

方案一：Spin WASM HTTP 应用

主流模式

业务代码是 HTTP server，由 Spin 托管为长驻进程，Dapr sidecar 双向通信。

外部请求 → Traefik (:80) → Dapr (:3500) → Spin WASM (:80)
                                ↕
                          Redis / Consul / ...
Nomad Job 含两个 task：spin-webhost + dapr-sidecar
支持 service invocation，可被其他服务调用
适用于 HTTP API、路由、微服务场景

方案二：Dapr WASM Binding

无 HTTP server

业务代码是 stdin/stdout CLI 程序，编译为 wasip1 WASM，由 Dapr 按需加载执行。

外部请求 → Traefik (:80) → Dapr (:3500) → bindings/wasm
                                ↕                → WASM (stdin→stdout)
                          Redis / Consul / ...
Nomad Job 只有一个 task：dapr-sidecar
没有 Spin 参与，WASM 产物上传到 dufs
适用于事件驱动、定时任务、轻量函数计算
